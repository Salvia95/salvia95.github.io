## 프로젝트 개요

2,300개 주유소 사업장에서 일 평균 89,000건 가량 발생하는 면세유류 매출이 POS 카드 승인과 PC 매출등록으로 이원화되어 있어, 사업장 근무자가 매일 54분을 면세유류 매출등록에만 할애해야 했습니다.

카드 승인 건에 대한 매출등록 누락으로 인해 매 분기 면세유류 환급금 신고액이 회계장부와 상이한 문제가 지속적으로 발생하고 있었고, 이를 해결하기 위해 업무를 재설계하고 자동매출 데몬을 개발했습니다.

## 왜 이 문제가 어려웠는가

### 실시간 처리가 불가한 구조

면세유류 매출은 일반 매출과 달리, POS에서 면세유 공급 승인을 먼저 받고 주유 후 결제하는 구조입니다. VAN사 응답전문에 면세유 공급가능 여부만 포함되어 있어 POS에서 실시간으로 매출등록을 요청하는 것이 불가능했습니다.

### 무거운 트랜잭션 구조

면세유 매출등록은 내부적으로 **과세유류 매입 반품 → 면세유 매입 → 면세유 매출**, 총 3개의 거래가 하나의 트랜잭션에서 이루어지는 특이한 구조를 가지고 있었습니다. 여기에 채권 처리까지 후행되어 건 당 평균 12.3초가 소요되고 있었습니다.

### 타 업무와의 자원 교착

매출 등록 라이브러리 실행 중 타 프로그램에서 반대 동작(채권 등록 ↔ 채권 회수)을 수행할 시 DB에 Deadlock이 발생하는 문제가 있었습니다.

## 해결 과정

### 업무 프로세스 재설계

기존 담당자의 PC 조작 중심 프로세스를 **스케줄 기반의 자동 실행**으로 재설계하여, 시스템 사용 숙련도에 따른 편차와 휴먼오류 발생 가능 지점을 제거했습니다.

### 검증 로직 전체 재개발

레거시에서 복제해온 코드에 도달할 수 없는 로직, 중복 로직이 다수 존재했습니다. 기존 30여 개의 검증 로직을 분해/재설계하여 **7개로 축소**하고, 회계 반영 가능 건만 먼저 필터링하는 순서로 변경하여 전체 수행 건 수를 크게 줄였습니다.

### 비동기 병렬처리 + 선택적 동기 실행

Spring Batch의 Chunk-based, asyncTask를 이용하여 전체 사업장의 매출 대상을 한번에 조회하고 청크 단위로 쓰레드에 분배했습니다.

- 지점 정보 기준 **모듈러 연산**으로 인스턴스별 작업 배정
- Fetch 시 **ReentrantLock**으로 중복 조회 방지 및 순서 보장
- 쓰레드 안전성이 확보되지 않은 외부 라이브러리에 한해 동기 수행을 보장

10개 시범 사업장에 대한 부하/처리량 테스트를 수행하여 **최적 인스턴스(2개) + 쓰레드(6개)** 환경을 도출했습니다.

### Deadlock 회피 전략

타 업무 라이브러리가 반대 동작 시 모든 대상에 Row Lock을 적용하는 것을 역이용하여, 잔액성 원장 수정 전 **Row의 Lock 여부를 확인하는 프로시저**를 구현했습니다. Lock 상태일 시 `SkipException`을 발생시켜 해당 건을 생략하고, 다음 실행 주기에 우선순위를 높여 재처리되도록 구성했습니다.

### 커스텀 AOP 기반 로그 서비스

배치 전환에 따른 관제 도구가 필요하여 LogWrapper 클래스를 구현했습니다. 메소드 실행부와 로그 기록부를 분리하고 `@Transactional(propagation = REQUIRES_NEW)`로 트랜잭션을 분리하여 로그 기록 실패가 매출 트랜잭션에 영향을 주지 않도록 설계했습니다.

## 주요 성과

| 지표 | 개선 전 | 개선 후 | 개선율 |
|------|--------|--------|------|
| 사업장 근로자 일 평균 업무 시간 | 54분 | 8분 | **86% 단축** |
| 매출 건 당 처리 시간 | 12.3초 | 0.97초 | **92% 개선** |
| 운영 안정성 | Deadlock/Timeout 빈발 | 12개월 무중단 | **완전 해소** |

## 활용 기술

- **Language**: Java
- **Framework**: Spring Framework, Spring Batch
- **Database**: Oracle RAC
- **Concurrency**: asyncTask, ReentrantLock, Chunk-based Processing
- **Monitoring**: Custom LogWrapper with Transaction Separation
