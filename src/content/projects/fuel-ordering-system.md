## 프로젝트 개요

전국 784개 가맹 사업장에서 일 평균 982건, 383만 리터(L) 가량 발생하는 연료제품의 수·발주 프로세스가 시스템 간 단절로 인해 수기 업무 중심으로 운영되고 있었습니다.

발주 및 배차 취소로 인한 영업 손실, 시스템 간 데이터 불일치로 인한 손익 분석 불가, 매입 부정행위 등의 문제가 지속적으로 발생하고 있었고, 이를 해결하기 위해 대외 시스템과의 연계 인터페이스 및 분산 트랜잭션 처리를 기반으로 서비스를 재설계했습니다.

## 핵심 문제와 해결

### 1. RESTful 할 수 없는 API 인터페이스

법인 API 게이트웨이를 경유해야만 대외 통신이 가능했으나, HTTP 헤더의 임의 조작이 불가하여 표준적인 인증 방식(Bearer Token 등)을 사용할 수 없었습니다.

**해결**: 요청/응답의 Body 내 공통부(Common)를 별도로 정의하고, **JWT 구조를 참고한 자체 토큰**을 구현했습니다. Header(알고리즘, 타입), Payload(발신자, 수신자, 타임스탬프, 요청 ID), Signature(HMAC-SHA256)로 구성하여 양측에서 동일한 Secret Key를 발행하고 무결성을 검증하도록 했습니다.

### 2. 분산 트랜잭션 처리

프로세스가 통합되면서 서로 다른 시스템에서 처리되더라도 중간 단계 실패 시 전체 트랜잭션의 롤백이 보장되어야 했습니다.

**해결**: Redis, Kafka 등이 없는 환경에서 **Oracle RAC을 이용한 Outbox 패턴**을 구현했습니다.
- RDBMS의 특징을 이용한 타임스탬프 기반 TTL 적용
- 오름차순 인덱스를 이용한 순차 처리
- 복합 키와 무결성 제약조건을 이용한 Set 형태 구현

### 3. 거래의 멱등성 보장

Outbound 거래에 대해서는 Outbox로 원자성을 보장했지만, Inbound 거래에 대해서는 중복 거래 식별이 불가했습니다.

**해결**: API 게이트웨이의 **프록시 서버를 내부 시스템에 구축**하고, 통신 정보와 데이터 공통부의 거래 ID/타임스탬프를 이용해 멱등성 키를 생성하여 **Inbox 패턴**을 구현했습니다.

### 4. 분산 환경에서의 Lock

배차 진행 중 발주 취소가 가능하거나, 이미 취소된 발주를 배차하는 등 트랜잭션 경계 문제가 발견되었습니다.

**해결**: RDB에 분산 락 테이블을 신설하고, 멱등성 키를 이용한 **명시적 락 + TTL** 전략을 적용했습니다. TTL 만료 시 담당자에게 카카오톡/문자 알림을 통해 즉시 보정 거래가 일어날 수 있도록 설계했습니다.

### 5. API 오케스트레이터

개별 서비스에 API 인터페이스 로직이 종속되어 보일러플레이트가 과도하게 늘어나는 문제를 해결하기 위해, **API 오케스트레이터 모듈**을 구현하여 모든 서비스가 내부 이벤트 버스를 통해 동일 트랜잭션 내에서 Outbox에 접근하도록 설계했습니다.

## 설계 원칙

이 프로젝트에서 가장 중요하게 여겼던 세 가지 원칙이 있습니다:

1. **기능은 배송사를 중심으로, 프로세스는 내부 시스템을 중심으로** - 각 주체의 자율성을 보장하면서 전체 프로세스를 관리
2. **새로운 기술보다는 기존 인프라를 최대한 활용** - 추가 자원 없이 기존 기술로 문제 해결에 집중
3. **무중단 상태에서의 점진적 전환** - 배송량이 낮은 지역부터 단계적으로 전환

- Oracle RAC 기반 In/Outbox 패턴 분산 트랜잭션 처리 프레임워크 구축

- API 오케스트레이터 및 콜백 기반 보상 트랜잭션 실행기 구현

- **Language**: Java

- **Database**: Oracle RAC
- **Pattern**: Outbox/Inbox Pattern, Distributed Lock, API Orchestrator, Callback-based Compensation
