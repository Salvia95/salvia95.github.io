## 프로젝트 개요

차세대 시스템 구축 후 2년이 채 되지 않은 시스템에서 레거시와 동일한 문제가 반복되고, 기술부채로 인해 개발 생산성이 급격히 저하되고 있었습니다.

시스템의 근본적인 문제점을 진단하고 조직의 맥락에 맞는 기술 스택을 선정하여, 변화의 타당성을 실증하기 위해 시작한 개인 프로젝트입니다.

## 문제 인식

운영 중인 시스템은 세 가지 근본적인 문제를 안고 있었습니다.

### 읽기/쓰기 책임의 미분리

복합키 기반의 단일 모델 아키텍처가 장기간 적용되면서 테이블 스키마가 과도하게 확장(평균 76개 컬럼)되었고, 쓰기 시에는 인덱스로 인한 성능 저하가, 읽기 시에는 적절한 인덱스 구성 불가로 성능이 하락하는 상황이었습니다.

### 모놀리식 아키텍처의 한계

비즈니스 로직과 도메인 모델이 강결합되어 기존 구현물의 변경이 사실상 불가능하고, 개별 UI를 기준으로 독립 구현된 서비스 구조로 인해 모듈의 응집도가 떨어져 단위 테스트 자체가 불가한 상태였습니다.

### 동기 호출 기반의 도메인 간 강결합

모놀리식 시스템의 동기 호출 특성으로 인해 개별 도메인에서 발생한 오류가 시스템 전체로 전파되고, 배포 시 의존성 문제로 전체 빌드 및 일괄 배포가 강제되는 상황이었습니다.

## 아키텍처 선택

### CQRS - 읽기/쓰기 DB 분리

능동적인 Scale Up/Out이 불가능한 Oracle RAC 기반 DB 클러스터의 근본적인 문제를 해결하기 위해 읽기/쓰기 역할이 분리된 **멀티 인스턴스 DB(PostgreSQL)** 구성을 선택했습니다.

- **쓰기 DB**: 정규화, 인덱스가 존재하지 않는 순수 쓰기 전용
- **읽기 DB**: 비정규화, 인덱스, 파티션으로 최적화한 조회 전용
- 5초 가량의 Eventual Consistency를 허용하는 업무 특성을 기반으로 분리

캐시 가능한 데이터가 전체의 2% 내외로 극도로 제한된 상황을 감안하여 Redis 클러스터 대신 **JPA 1차 캐시와 기준정보성 데이터에 대한 2차 캐시(인스턴트 로컬)** 를 활용하는 전략을 채택했습니다.

### 모듈러 모놀리스

도메인 컨텍스트의 식별, CI/CD 파이프라인, 테스트 환경 등이 갖춰지지 않은 상태에서 MSA로의 즉시 전환은 오버엔지니어링이라 판단했습니다.

모놀리식 시스템의 간결함을 유지하면서 모듈 간 독립성을 확보할 수 있는 **Spring Modulith**를 구현체로 선정하여:

- 모듈 영역을 강제하고 의존성 관계를 명확히 정리
- DDD에 입각한 **POKO(Plain Old Kotlin Object)** 기반 도메인 모델링으로 물리적/기술적 외부 독립성 보장
- 클린 아키텍처를 참고한 모듈 내 계층 위계 적용, **Facade/Gateway 패턴**으로 모듈의 입출력부를 추상화

### 비동기 이벤트 통신

시스템 규모(평균 760TPS, 10만 MAU)와 B2B 업무 특성, 추가 인프라 구성 불가 상황을 고려하여 Kafka 대신 **Spring Modulith의 Event Publication Repository를 이용한 RDBMS 기반 Outbox 이벤트 소싱**을 선택했습니다.

Eventual Consistency를 허용하지 않는 중요 작업(주문 후 재고, 결제 연계 등)에 한해 동기 실행 형태의 이벤트 체인을 예외적으로 허용하는 하이브리드 방식을 적용하고 있습니다.

## 활용 기술

- **Language**: Kotlin
- **Framework**: Spring Boot, Spring Modulith
- **Database**: PostgreSQL (Multi-Instance), JPA
- **Architecture**: Modular Monolith, CQRS, DDD, Clean Architecture
- **Event**: RDBMS-based Outbox Pattern, Local Event Bus
