## 들어가며

IT 조직에서 처음으로 지식 베이스를 구축해야할 때 전문적인 테크니컬 라이터 없이 높은 품질의 문서 관리체계를 설계하고 관리할 수 있을까요?

제 생각은 "불가능하다" 였습니다. 그렇다면 테크니컬 라이터가 없는 조직은 문서 관리체계의 구성을 포기해야할까요?

이 문서는 이전 이야기인 [Obsidian으로 구성해본 폐쇄망 환경에서의 클라이언트 중심 Wiki 페이지!](/posts/obsidian-wiki)를 발전 시키기 위해 생성형 AI의 도움을 받아 전문적인 인력 없이 개인이 조직 차원에서 활용할 수 있는 지식 베이스를 구축한 경험을 다룹니다.

## 구현 후 있었던 일

Obsidian을 이용해 작은 위키 페이지를 구성하고 지속적으로 문서를 작성하고 관리하면서 간혹 꼬여버리는 브랜치 문제, 잘못된 정보의 유입 문제도 있었지만, 가장 큰 문제는 "일률적이지 못한 양식과 문서의 품질" 이었던 것 같습니다.

Template와 같은 플러그인을 사용하면서 기술 문서 작성을 위한 기준을 정하고 이를 활용하여 문서를 작성하도록 한다면 통일성 있는 문서를 작성할 수 있을 것이라 기대했었지만, 실제로 마주한 현실은 기대와는 크게 달랐습니다.

전체적인 틀은 공유를 하지만, 문서 내부에 작성되는 글의 경우 작성자의 성향에 따라 문체와 용어가 통일되지 않아 문서의 품질이 균일하지 못했고, Obsidian 활용 수준에 따라 링크가 잘못되거나, 그림 등의 리소스 관리가 적절히 이루어지지 않아 문서가 깨져버리는 일도 종종 발생하였습니다.

무엇보다 Wiki의 가장 큰 장점인 참여자를 통한 수정이 많은 사용자를 확보하지 못한 상황에서는 오히려 단점으로 작용하여 Wiki 전체의 지식이 소수가 가지고 있는 지식으로 편향되고 있는 상황이었습니다.

## 이제 혼자는 더 할 수 있는게 없는데..

당찬 목표를 가지고 시작한 Wiki 페이지의 운영이었지만, 모든 일들이 원하는 방향으로 흘러가지 않고 오히려 원하는 바와 정 반대의 방향으로 흘러가고 있는 것을 보며 약간의 좌절감을 느꼈던 것 같습니다.

본래 각자가 혼자 작성하고 있던 문서들을 모아놓았을 뿐 여전히 많은 사람이 사용하기에는 초기 문서가 부족한 상황이었고, 간간히 추가되는 지식마저 파편성 지식에 가까워 실질적은 규모의 확장도 멈춰버린 상황이었기에 더 이상 개인의 힘으로 할 수 있는 것이 없는것 처럼 느껴졌습니다.

## 꼭 사람이 해야하는 건 아니잖아?

Wiki의 운영 유지에 대해 많은 생각을 하던 중 사내 생성형 AI 활용을 위한 테스트가 진행 중이라는 소식을 듣게 되었고 이를 통해 한 가지 아이디어가 떠올렸습니다.

"생성형 AI를 테크니컬 라이터로 사용하는 건 어떨까?" 프롬프트를 통해 생성형 AI의 역할과 수준을 잘 지정한다면, 데이터만 확보되면 일정한 수준의 문서를 끊임없이 자동으로 생성할 수 있다는 생각이 들었기에 이에대한 아이디어를 구체화 해보기로 했습니다.

## 아이디어 구체화 해보기

Wiki에 문서를 작성하기 위해 수행했던 행위를 분류해보자면 본다면 크게 문제 해결을 통한 지식의 습득 과정과 습득한 지식을 간략한 글로 정리하는 과정, 정리된 내용을 Wiki의 문서 작성 양식에 맞춰 재가공하는 과정으로 크게 3개의 단계로 나눌 수 있을 것 같습니다.

이 중 지식의 습득 과정과 습득한 지식의 정리 과정의 경우 업무를 진행하며 수시로 이루어지고 있던 작업이었기에 오랜 시간이 소요되지 않았지만, 습득된 지식을 간략히 정리한 문서를 Wiki에 등재하기 위한 문서로 재가공하는 과정은 기존에 많이 해보지도 않았거니와,

## 이게 되네?

구체화된 아이디어를 검증하기 위해 샘플을 만들고 개인적으로 생성형 AI(Claude)와 Github MCP를 통해 테스트 해 본 결과 Wiki 내 템플릿을 읽고 양식에 맞춰 마크다운 문서를 작성할 수 있다는 것을 확인하였고, 약간의 수정은 필요하지만 생각보다 높은 수준의 문서를 결과물로 얻을 수 있었습니다.

이를 통해 생성형 AI를 문서 작성의 주체로 활용할 수 있다는 사실을 알 수 있게되었고 이를 활용한다면 문서작성에 들어가는 공수도 꽤나 크게 줄어들 것이라는 기대도 하게되었습니다.

하지만, 사람의 욕심은 끝이 없다고 했던가요? 생성형 AI를 이용해 문서 작성을 자동화한 것만 해도 큰 발전이었지만, 되는 걸 확인하니 이제는 문서의 생성 자체를 자동화하는 것 또한 가능하지 않을까 라는 생각을 하게되었습니다.

## 생각해보니 나는 아는 것도 가진것도 없었다

생각만 해도 즐거운 상상의 뒤에 찾아온 것은 밝은 미래에 대한 희망 아닌 현실에 벽에 부딪히는 막연한 두려움이었습니다.

설계를 통해 이론적 상으로는 실행이 가능한 프로세스를 정의했지만, 문제는 이를 구현하는 일이었습니다.

주로 시스템 내 서비스의 개발과 운영에 초점을 맞춰 프론트엔드 혹은 백엔드 개발자의 역할을 수행하며 지식과 경험을 쌓았지만, 공공 혹은 금융권의 특수한 환경적 요인으로 인해 정작 생성형 AI의 활용과 성능 측정에 대해서는 경험해본 바 없고, 가진거라곤 학부와 석사과정 중 수강한 기본적인 데이터 과학이나 머신러닝에 대한 지식밖에는 없는 상황이었습니다.

## 아무것도 몰라도 된다?!

구현을 위한 방법을 고민하던 도중 언젠가 유튜브와 여러 기업들의 테크블로그에서 보았던 `n8n`, `Zapier`, `Make` 등의 워크플로우 자동화 도구들이 떠올랐습니다.

위와 같은 도구의 특징은 워크플로우와 관련된 사전 지식이 없는 사람이라도 **특별한 학습없이 생성형 AI를 포함한 워크플로우를 간단히 구현할 수 있다는 것**으로 공통적으로 약간의 개발 지식이 있다면 Edge Function을 통해 조금 더 세련된 워크플로우를 구축할 수 있다는 것을 장점으로 내세우고 있었습니다.

그 외에도 사전 정의노드, 워크플로우 템플릿 등 다양한 장점들이 존재했지만, 백문이 불여일견이라 했던가요? 직접 확인해보는 것 만큼 빠르게 이해할 수 있는 수단은 없다고 생각했기에 더 이상 고민하지 많고 빠르게 간단한 테스트를 수행해보기로 했습니다.

## 어떤 툴이 적합할까?

조직 차원에서의 운영은 라이센스가 비교적 자유로운 `Airflow`를 사용하여 커스터마이징을 하거나 `Zapier`를 사용하여 신뢰도를 높이는 방향으로 진행되는 것 같았지만, 복잡한 테스트 과정 등으로 인해 개인이 운영 가능한 서비스를 만드는데는 적합하지 못하다는 생각이 들었습니다.

그래서 최종적으로 사용하기로 한 워크플로우 자동화 툴은 `n8n`으로 **대부분의 노드가 사전 정의되어 간단한 설정만으로도 사용이 가능**했고, Self-hosting 시에도 비교적 적은 컴퓨팅 자원을 활용했으며, 무엇보다 하나의 컨테이너로 설정을 위한 웹페이지와 노드의 사용이 가능했기에 가장 적합하다고 판단했습니다.

## 간단한 테스트 진행해보기

간단한 테스트를 위해 `n8n` 컨테이너를 도커 상에서 동작시키고, 호스팅된 웹페이지에서 노드의 구성과 간단한 설정을 진행하였습니다.

테스트 과정에서 사용한 노드는 Webhook 노드와 AI Agent노드로 각 각은 데이터의 수집과 수집된 데이터를 통해 마크다운 문서를 작성 동작을 주 역할로 하고있으며, 예상된 프로세스는 아래와 같습니다.

1. 데이터의 발생 및 전달(Server에서 Webhook 발생)
2. 데이터 수신 및 분석 요청(Webhook으로 AI Agent Trigger)
3. 데이터 분석 및 문서작성(AI Agent로 LLM에 전달)
4. 작성된 문서 저장(AI Agent로 MCP를 이용해 임시영역에 저장)

실제 서비스에는 더 상세한 프로세스가 필요하겠지만, **현 단계에서는 가장 핵심이 되는 부분만 선정하여 테스트 프로세스를 구성**하였으며, 이를 테스트하기 위한 프롬프트는 아래와 같이 구성하였습니다.

```markdown
당신은 IT 조직의 테크니컬라이터로 기술적 이슈사항을 문서로 작성하고 시스템을 운영하는 담당자가 향후 이를 참고하여 업무를 수행할 수 있도록 돕고 있습니다.

데이터는 JSON 형식으로 이루어져 있으며, 메타데이터에 해당하는 header부와 실제 데이터인 body부로 나누어져 있습니다.
다음 데이터를 분석하여 기술적 이슈사항을 확인하고 적절한 메타데이터를 추출하여 태그를 구성, 문제와 해결책을 마크다운 형식의 문서로 변환하십시오
변환이 완료된 마크다운 파일은 MCP를 이용해 MD파일로 저장되어야 합니다.

## 데이터:
{{ $json.data }}

응답은 Obsidian에서 즉시 열어볼 수 있는 MD 파일 형식으로 작성해야하며, 다음의 형식을 준수하여야 합니다.

--- 문서 시작

1. **태그**
2. **문제의 정의**
3. **기술적 원인**
4. **해결 방안**: (기술적 원인을 해결할 수 있는 최적의 방안)
5. **각주**: (신규 투입된 개발자가 모를 수 있는 용어와 시스템 개념)

--- 문서 종료
```

데이터를 발생시킬 서버가 존재하지 않는 상황이기에 샘플 데이터를 Webhook 노드에 직접 입력하였으며, 입력된 JSON 형식의 데이터는 아래와 같습니다.

```json
{
  "header": {
    "issueData": "20250705",
    "issuedBy": "Q&A",
    "issuedSystem": "생산경제시스템"
  },
  "body": {
    "List": [
      {
        "subject": "사용방법",
        "question": "화면 내 엑셀 기능이 동작하지 않아요",
        "answer": "Window11로 업데이트를 진행하신 후 엑셀이나 한셀 등이 설치되지 않은 경우 혹은 xlsx 파일의 기본 형식이 등록되지 않은 경우 엑셀 기능이 정상적으로 동작하지 않을 수 있습니다. 해당 기능을 점검하신 후 정상동작하는지 확인하여 주시기 바랍니다. 감사합니다."
      }
    ]
  }
}
```

워크플로우의 실행은 Webhook 트리거로 시작되며, LLM을 통해 생성된 응답은 MCP를 이용해 MD 파일로 생성되었으며, 생성된 MD 파일의 내용은 아래와 같습니다.

```markdown
# 생산경제시스템 엑셀 기능 오류 해결 가이드

## 1. 태그
`#생산경제시스템` `#엑셀기능` `#Windows11` `#파일연결` `#사용자지원` `#시스템호환성` `#오피스소프트웨어`

## 2. 문제의 정의
생산경제시스템 내에서 엑셀 관련 기능이 정상적으로 동작하지 않는 문제가 발생하고 있습니다.
사용자가 화면 내에서 엑셀 기능을 실행하려고 할 때 응답하지 않거나 오류가 발생하는 상황입니다.

**발생 환경:**
- 시스템: 생산경제시스템
- 운영체제: Windows 11 업데이트 후
- 문제 유형: 엑셀 기능 비활성화

## 3. 기술적 원인
Windows 11로 운영체제를 업데이트한 후 발생하는 문제로,
다음과 같은 기술적 원인들이 확인되었습니다:

### 3.1 오피스 소프트웨어 미설치
- Microsoft Excel 또는 한글과컴퓨터 한셀 등의 스프레드시트 소프트웨어가 시스템에 설치되지 않음
- 시스템이 엑셀 파일을 처리할 수 있는 응용프로그램을 찾지 못함

### 3.2 파일 연결 설정 오류
- .xlsx 파일 확장자에 대한 기본 프로그램 연결이 해제되거나 손상됨
- Windows 레지스트리에서 파일 형식과 응용프로그램 간의 매핑 정보 손실
- COM 객체 등록 정보 오류로 인한 프로그램 간 통신 장애

## 4. 해결 방안

### 4.1 즉시 조치 사항
1. **오피스 소프트웨어 설치 확인**
   제어판 > 프로그램 및 기능에서 Microsoft Office 또는 한셀 설치 여부 확인
   미설치 시 조직 표준 오피스 소프트웨어 설치 진행

2. **파일 연결 재설정**
   설정 > 앱 > 기본 앱 > 파일 형식별 기본 앱 선택
   .xlsx 파일을 적절한 스프레드시트 프로그램으로 연결

### 4.2 시스템 관리자 조치 사항
1. 조직의 소프트웨어 배포 정책 점검
2. Windows 11 호환성을 고려한 오피스 버전 배포
```

## 새로운 시작

Wiki의 운영 문제에 대해 인식하고 생성형 AI를 이용한 해결방안을 도출, 구현을 위한 워크플로우 자동화 툴 선정과 간단한 테스트를 통한 구현 가능성까지 진행된 현재 "과연 이게 가능할까?"로 시작된 의문이 "충분히 현실이 될 수 있겠다!" 라는 생각으로 바뀌기 시작하였습니다.

특히, 생각보다 학습 곡선이 더 낮고, 필요한 기능이 대부분 노드의 형태로 정의된 상태로 존재하고 있다는 것을 확인하였기에 이를 바로 실제 서비스로 구현하는 과정에 들어갈 수 있었으며, 지체하지 않고 바로 프로토타입의 제작으로 넘어갈 수 있는 원동력이 되었던 것 같습니다.
